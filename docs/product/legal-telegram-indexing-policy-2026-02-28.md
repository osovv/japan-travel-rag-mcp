# Telegram Indexing Legal Playbook (RU, Solo Founder, Draft)

Date: 2026-02-28
Project: `travelmind-mcp`
Status: Working draft (not legal advice)

## 1. Контекст и факты по текущей реализации

1. Продукт: не AI/ML-платформа, а индекс + поиск по Telegram-сообщениям.
2. Текущий скоуп источников:
   - `Source A`: публичный авторский канал.
   - `Source B`: публичный чат путешественников при канале.
3. Публичный MCP tool surface сейчас ограничен 4 методами:
   - `search_messages`
   - `get_message_context`
   - `get_related_messages`
   - `list_sources`
4. `TG_CHAT_RAG_CHAT_IDS` уже используется как allowlist и принудительно инжектится в `search_messages`.
5. Отдельного API/tool для «полной выгрузки истории чата» в текущем surface нет.
6. Открытый техдолг: локальная проверка chat scope для message-based методов (`get_message_context`, `get_related_messages`, `list_sources`) пока не доведена до такого же уровня, как для `search_messages`.

## 2. Главный ответ на вопрос «это вообще законно?»

Коротко: в РФ это обычно возможно, если делать аккуратно.

1. Сам факт, что канал/чат публичный, не отменяет правовых обязанностей.
2. Риск здесь в основном не в «чтении», а в вашем повторном использовании данных в сервисе (индексация, хранение, выдача).
3. Для старта в РФ рабочая модель: узкий allowlist, минимизация выдачи, прозрачные правила удаления, без массового реэкспорта контента.

## 3. Нужно ли получать чье-то approval

1. Для собственного авторского канала (`Source A`): достаточно явного решения правообладателя (по сути, вашего).
2. Для публичного чата сообщества (`Source B`): желательно письменное согласие админов/владельца чата (короткий email/сообщение с разрешением на индексацию и поиск).
3. Если по `Source B` нет письменного подтверждения:
   - запуск с `Source A` возможен;
   - `Source B` лучше держать выключенным до подтверждения.

## 4. Какие персональные данные реально могут быть в публичном чате

Даже в открытом чате часто встречаются:
1. `@username`, display name, Telegram user ID.
2. Контакты (телефон, email, ссылки на личные профили).
3. Геометки, даты поездок, маршруты, место проживания.
4. Связки фактов, по которым человека можно идентифицировать косвенно.

## 5. Почему ответственность есть даже при публичном источнике

1. Вы становитесь отдельным оператором/обработчиком своей базы индекса.
2. Вы меняете режим доступа: поиск по вашему API может упростить извлечение данных.
3. Поэтому ответственность возникает из вашей обработки и выдачи, а не из факта, что сообщение уже видно в Telegram.

## 6. Риск-модель без нагнетания

Нормальный рабочий профиль риска для solo founder:

1. Низкий риск:
   - 1-2 allowlisted источника,
   - нет bulk-export,
   - короткие snippets,
   - быстрый takedown.
2. Средний риск:
   - индекс общественного чата без письменного разрешения,
   - слабая процедура удаления,
   - широкая выдача контента.
3. Высокий риск:
   - массовое подключение чатов без контроля,
   - инструменты для реэкспорта больших массивов сообщений,
   - игнорирование жалоб/удалений.

## 7. Минимальный legal-пакет для запуска (без тяжелой бюрократии)

1. `Privacy Policy` (короткая, по факту вашей обработки).
2. `Terms of Use` (ограничение ответственности, запрет злоупотреблений).
3. `Source Registry` (внутренний файл: source, основание, дата одобрения, статус).
4. `Takedown Policy` (куда писать, что прислать, SLA ответа).

## 8. Что уже ок и что надо доделать

### Уже ок
1. Allowlist через `TG_CHAT_RAG_CHAT_IDS` в конфиге есть.
2. Принудительный `chat_ids` policy path для `search_messages` есть.
3. Публичный tool surface ограничен.

### Обязательный ближайший фикс

2. Прописать и опубликовать простой takedown-канал (email/форма).

### Желательно, но не блокер первой поставки
1. Строже ограничить размер выдаваемых фрагментов в ответах.
2. Добавить короткий публичный changelog по policy-обновлениям.

## 9. Правило запуска для текущего кейса

1. `Source A`: можно оставить в проде.
2. `Source B`: 
   - есть письменное согласие админов -> можно включать,
   - нет согласия -> выключить и не индексировать до получения.

## 10. Быстрый runbook при жалобе

1. Принять запрос и зафиксировать время.
2. Найти записи по `message_uid`/`chat_id`.
3. Удалить из индекса + очистить производные кэши.
4. Подтвердить заявителю выполнение и срок.
