# Telegram Indexing — Практичный План Для Solo Founder (Draft)

Date: 2026-02-28
Project: `japan-travel-rag-mcp`
Status: Рабочий черновик (не юридическая консультация)

## 1. Зачем этот документ

Этот документ нужен, чтобы:
1. Понять, что реально обязательно сделать до запуска.
2. Не тратить время на лишнюю юридическую бюрократию.
3. Запуститься в более безопасном режиме и не словить лишние проблемы.

## 2. Простыми словами: в чем суть

1. Публичный чат можно читать вручную, но сервис с индексом и поиском — это уже отдельная обработка данных.
2. Это не катастрофа и не "запрет на запуск", это вопрос правильных ограничений и базовой гигиены.
3. Для solo founder рабочая стратегия: узкий скоуп + явные правила + быстрый процесс удаления.

## 3. Текущий рабочий скоуп

1. `Source A` — публичный авторский канал: `GO` при базовых условиях.
2. `Source B` — публичный чат путешественников: `GO` только при письменном разрешении админов чата.
3. Если разрешения по `Source B` нет — запускаемся только с `Source A`.

## 4. Минимум, который нужно сделать до production

1. Источники:
   - Явный allowlist источников (только утвержденные `source_id`).
2. Выдача:
   - Только snippet-first.
   - Без дампа полной истории сообщений.
3. Документы:
   - Короткие `Privacy Policy` и `Terms`.
   - Публичный контакт для `takedown/удалений`.
4. Удаления:
   - Удаление по `message_uid` + очистка кешей.
   - Журнал удалений (дата, запрос, результат).

## 5. Что можно отложить (не блокирует первый запуск)

1. Сложная enterprise-процедура комплаенса.
2. Большая формализация governance.
3. Продвинутый legal-automation и тяжелые процессы согласования.

## 6. Красные линии (чего не делать)

1. Не индексировать `Source B` без письменного разрешения админов.
2. Не делать full-history export endpoint.
3. Не игнорировать запросы на удаление/жалобы.
4. Не расширять источники автоматически без обновления allowlist.

## 7. План на 7 дней (реалистично для одного человека)

### День 1
1. Зафиксировать allowlist и текущий скоуп (`A` + условно `B`).
2. Принять правило: нет разрешения по `B` -> `B` выключен.

### День 2-3
1. Опубликовать короткие `Privacy Policy` и `Terms`.
2. Добавить email/форму для takedown и удаления данных.

### День 4-5
1. Внедрить удаление по `message_uid`.
2. Добавить purge производных кешей.

### День 6
1. Проверить, что выдача только snippet-first.
2. Проверить, что full-history выгрузка отсутствует.

### День 7
1. Прогнать чеклист `GO/NO-GO`.
2. Запустить в ограниченном режиме и мониторить обращения.

## 8. Чеклист GO/NO-GO перед запуском

`GO`, если:
1. Источники ограничены allowlist.
2. `Source B` имеет письменное разрешение или выключен.
3. Есть публичный канал для takedown/удалений.
4. Есть per-message удаление (`message_uid`) и purge кешей.
5. Выдача snippet-first, без full-history экспорта.

`NO-GO`, если хотя бы один пункт выше не выполнен.

## 9. Если пришла жалоба — быстрый runbook

1. Зафиксировать запрос (кто, что, когда).
2. Удалить запись по `message_uid` из индекса.
3. Очистить кеш/производные данные.
4. Подтвердить заявителю результат и время обработки.
